Findings from adversarial review of the three design documents against `d:\Math\self\cayleypy\conjugacy_bfs_characters\notes\character_method_cayley_diameters.md`.

## Findings

**1. BLOCKER — Docs 1 and 2 describe two incompatible engines; the plan cannot implement "both".** The divergences are load-bearing, not cosmetic: (a) certification — Doc 1 §7.4 runs a synchronous tiered gate (bound / resident-CRT / dynamic-extra-primes / bigint) *every radius* and explicitly demotes bigint to tier 4; Doc 2 §5–6 explicitly *rejects* CRT as primary and certifies asynchronously via the monotonicity lemma with 1–2 bigint evaluations at `d−1/d−2` per type; (b) stopping — Doc 1 stops on certified empty layer (spec §5.2) with a parity-bound early exit; Doc 2 makes empty-layer a *hard error* and terminates only on `visited == predicted reachable set`; (c) transpose pairing — Doc 2 §4.2 halves table storage/GEMM (`R½` rows, `W±`, parity-ordered targets); Doc 1's `TableBlock`/`TransformBackend`/memory table have no pairing at all (n=50 table: Doc 1 says 500 GB @ k=3; Doc 2 says 167 GB @ k=2 paired); (d) arithmetic regime — Doc 1: 3×31-bit primes, u128 accumulate; Doc 2: 2×30-bit primes, lazy-16 u64 + Barrett; Doc 3 §5.3 tests Doc 1's regime; (e) backend trait — Doc 1 `TransformBackend {init, begin_radius, apply_block, end_radius}` (push-based blocks, no parity split) vs Doc 2 `ModTransformBackend {load_table, gemm(targets: Range), gemm_multi}` (parity-aligned target ranges, per-(n,prime) instance); (f) checkpoint payload — Doc 1: first-hit arrays only, powers recomputed by modexp, written only post-certification; Doc 2: powers `P,P′`, exact powers, and a pending-certification queue serialized. **Fix**: the plan must adopt one merged design per component. Recommended merge: Doc 1's stopping rule + per-radius certification gate + checkpoint-recompute + result schema; Doc 2's transpose pairing, `W±` bookkeeping, and cost model; both certifier styles retained but with Doc 1's tiers 1–2 as the free screen and Doc 2's shared-exact-column batching as the fallback tier; pick ONE prime regime (31-bit/u128 on CPU is simpler and proven — 204226 · (2³¹)² < 2¹⁸·2⁶² = 2⁸⁰ ≪ 2¹²⁸; keep 30-bit/u64 as the documented GPU regime only) and one backend trait shape.

**2. MAJOR (soundness) — Doc 1 contains a mathematically false assertion that will abort valid runs.** §7.2: for a type whose same-parity first hit is already recorded, "all-zero residues for such a type would expose a backend bug (asserted)". Wrong: `a_r(ν) > 0` with `a_r(ν) ≡ 0` mod every resident prime is possible (probability ≈ 2⁻⁹³ naturally, but **deterministically triggered** by Doc 3 §5.1's injected small primes {11,13} on already-visited types — the adversarial suite would trip this assert constantly). The step-4 comment "cannot happen: r+2 lemma guarantees a nonzero residue is mathematically present" confuses a nonzero *integer* with a nonzero *residue*. **Fix**: classify such types POSITIVE by the lemma alone (which Doc 1 already does), count all-zero-residue occurrences in a metric, never assert on them.

**3. MAJOR (soundness/completeness) — Doc 2's certification scheme under-certifies the required output.** The `d−1/d−2` two-evaluation argument certifies *distances*. The user requires exact per-radius supports; for mixed-parity unions the reconstruction `support_r = {ν : fh[r mod 2][ν] ≤ r}` needs BOTH per-parity first hits certified, i.e. up to two zero-evaluations *per parity chain* (up to 4/type), and the opposite-parity first hit `r₁ > d` additionally needs `a_{r₁−2}(ν) = 0` certified. Doc 2 mentions `d−3` only in the repair path. Doc 1's every-radius gate delivers this automatically. **Fix**: if the async lemma-based certifier survives into the plan, its obligation set must be "both per-parity first hits", not "the distance".

**4. MAJOR — Doc 2's termination rule inverts the spec's correctness posture.** It *requires* the theory prediction (normal closure ⊇ A_n for n ≥ 5) to terminate and treats spec §5.2's empty-layer stop as an assert-only hard error, needing a special brute-force path for n ≤ 4 and identity inputs. The theorem is true, but spec §2.2 says shortcuts "must not affect correctness" and §5.2 designates the empty-layer rule. Doc 1's arrangement (empty-layer primary; `AllTypesVisited` early exit justified purely as a parity *upper bound* — sound at every n, including `[2,2]` in S₄/V₄) is the correct one. **Fix**: adopt Doc 1's stopping; keep Doc 2's prediction as an always-on cross-check assert.

**5. MAJOR — checkpoint granularity vs the n=50 certification tail can livelock across 2 h sessions.** Doc 2 §6.3 prices n=50 huge-class certification at 1.5–2 h per layer tail and itself flags "must checkpoint mid-certification". Doc 1 checkpoints only *committed* (fully certified) layers and discards everything else on deadline — so any (transform + certify) unit exceeding the remaining wall is redone forever on resume. **Fix**: allow an intra-layer checkpoint containing the layer's screened positives + certified-so-far candidate verdicts + remaining queue, with the invariant that the stopping test and layer commit read only fully-certified state; on resume, re-enter certification, never skip it. (Doc 1's current design is otherwise the correct answer to review question 4: a mid-layer kill discards the uncommitted radius, so a checkpoint can never encode an uncertified stop — CONFIRMED for n ≤ 45.)

**6. MAJOR — eigen-grouped mode is inexpressible in Doc 1's batched backend API.** `apply_block` computes `out[t,j] = Σ_r data[t·R + r]·w[r,j]` with one row space shared by *all* union columns; eigenvalue groups are per-union (`E_j` differs, group membership differs), so a shared `TableBlock` cannot carry per-union `H_α` planes. Doc 2's design instead accumulates per-union `H` arrays and does per-union small GEMVs — a different dataflow. **Fix**: specify eigen-grouped execution as a per-union (batch = 1) path or a separate `GroupedTransform` entry point; do not pretend `EigenGroupedSource` fits the shared-row `CharacterSource` contract ("row semantics opaque" breaks precisely at the column-shared weight matrix).

**7. MAJOR — Doc 2 as reviewed is truncated.** It begins mid-sentence inside §4; §§0–4.1 — including the cited generation-cost formula (§2.2), the exact-column trie mode (§2.3 "mode (b)"), and target-parity column grouping (§3.3) — are absent. Several cost claims (T_gen ≈ (2–4)·p(n)², "15 s/sweep at n=50") rest on the missing text. **Fix**: recover/rewrite the missing sections before planning; do not cite them sight-unseen.

**8. MAJOR — hardware assumption is never pinned, and the whole mode ladder hangs on it.** Resident n=45 (32 GB paired k=2 / 95 GB unpaired k=3) and stored n=50 (167 GB) assume a ~300 GB, 96-core box. Kaggle CPU/GPU notebooks are ~4–16 vCPU / ~30 GB; only TPU-VM instances have ~96 vCPU / ~330 GB. Doc 2 itself cites "~70 GB disk". Both docs *do* gate residency at runtime (good), but the phase targets, k choice, and time table silently assume the big machine. **Fix**: name the target tier explicitly in the plan; recompute the n=45/50 mode selection for the actual RAM; keep runtime verification authoritative.

**9. MINOR — Doc 2's H_α numbers are internally off ~2×–4×.** (a) Wave memory: its own formula `|E|·q·4·k` per union gives 2451·204226·4·2 ≈ 4.0 GB/union → 20 unions = **80 GB**, stated as "40 GB" (k factor dropped). (b) H-build cost is scatter-bound, not MAC-bound: R½·q·m·k ≈ 102k·204226·100·2 ≈ 4.2×10¹² scatter-adds at ~2.4×10¹⁰/s aggregate (3–6 ns/op × 96 cores) ≈ **3 min**, not "≈ one dense GEMM radius" folded into "15 s/sweep × 5 waves" (the MAC rate 1.4×10¹¹/s is ~6–15× the scatter rate). End-to-end conclusion (certification-dominated, ~30–45 min) survives, but fix the table. Other headline numbers **check out**: table bytes q²·4 = 126 MB / 886 MB / 5.58 GB / 31.8 GB / 166.8 GB at n = 30/35/40/45/50 ✓; n=45 GEMV 3·89134² ≈ 2.4×10¹⁰ MACs ≈ 0.2 s ✓; fused sweep 102k·204226·75·2 ≈ 3.1×10¹² MACs ≈ 22 s ✓ (result buffer is 61 MB at r_max=75, not 49 MB — trivial).

**10. MINOR — GPU-regime overflow bound is stated at exactly the overflow boundary, and the f64 fallback is vacuous.** "16·2⁶⁰ = 2⁶⁴" does not fit u64 as written; the true bound is 16·(2³⁰−1)² + p < 2⁶⁴ − 2³⁵ + 16 + 2³⁰ < 2⁶⁴ — safe, but only with `p ≤ 2³⁰ − 1` strict and the carried reduced partial included. Doc 2's wgpu alternative "f64 with block ≤ 2⁵³/p²" gives block < 1 at p ≈ 2³⁰ (2⁵³/2⁶⁰ = 2⁻⁷): f64 GEMM requires p ≲ 2²⁶. **Fix**: restate both bounds precisely in the backend contract; drop or re-derive the f64 option.

**11. MINOR — Doc 3's clippy `disallowed-types = [f32,f64]` in core, as specified, fails the build.** Doc 1 puts `orchestrate::estimate` (`RangeF64`, `est_prepare_s`), `timings_s`, `elapsed_before_s: f64`, and `Calibration` inside core. **Fix**: scope the float ban to the math modules (`arith`, `chars`, `spectra`, `transform`, `engine`) or move estimation/timing types to the CLI crate.

**12.
