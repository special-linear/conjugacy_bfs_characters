# Numerics & Algorithms Design: Character-Method Cayley Diameters (Rust Core)

**Scope of this document:** algorithm-level design of the Murnaghan–Nakayama (MN) evaluator, modular kernel, exact-integer strategy, certification path, large-`n` strategy, cost model, and GPU-backend contract. Architecture/crate-layout/API details are assumed from the fixed project decisions and covered only where numerics force a choice.

**Notation:** `q = p(n)` (number of partitions), `R` = active irreducible rows, `R½ ≈ q/2` = transpose-paired representative rows, `k` = number of screening primes (default 2), `m` = number of unions in a batch, `r` = radius, `X[ρ,ν] = χ^ρ(ν)`.

Reference values: p(30)=5,604 · p(35)=14,883 · p(40)=37,338 · p(45)=89,134 · p(50)=204,226.

---

## 0. Executive summary and regime map

The design has **one algorithmic core** (modular column-generated character transform + spectral powering + exact certification) operated in three storage regimes:

| Regime | n | Table strategy | Why |
|---|---|---|---|
| A: stored table | n ≤ ~48 | Full paired half-table in RAM per prime (u32 residues) | 2 primes × q²/2 × 4 B = 126 MB (n=30), 5.6 GB (n=40), 31.8 GB (n=45), 86 GB (n=48) — fits 300 GB easily |
| B: stored, tight | n = 49–50 | Same, k=2 only | n=50: 2 × 83.4 GB = 167 GB — fits a 300+ GB Kaggle VM, but no headroom for 3 primes |
| C: streamed | n ≥ 50 (robust path), or low-RAM | **Never store the table.** One generation sweep per union-wave, aggregated either into eigenvalue blocks `H_α` (preferred, when `|E_U|` is small — true for the primary use case of bounded-support classes) or fused across all radii at once | table regeneration (~10 s) is cheaper than reading it from any disk; Kaggle's ~70 GB disk cannot hold the n=50 table anyway |

Three load-bearing conclusions that differ from the preliminary analysis, argued in detail below:

1. **Exact fallback is not rare — it is structural.** Every finite distance assignment `d(ν)=r` requires certifying `a_{r−1}(ν)=0` and `a_{r−2}(ν)=0` exactly (a monotonicity lemma reduces "all r′<r" to just these two; parity kills one of them for single-parity unions). So the exact path runs `~1–2 × (reachable types) × (unions)` times per run, not "a handful of times." It is still affordable (seconds to minutes; §6), but it must be designed as a first-class batched subsystem, not an afterthought. Primes only prevent *spurious* certifications of positive entries; exactness never depends on `k`.
2. **Stopping should not be detected, it should be predicted.** For `n ≥ 5` the subgroup generated by a nonempty conjugation-invariant set (excluding identity-only) is normal, hence `A_n` or `S_n`. The reachable type set is therefore known a priori (all even types, or all types), and the run terminates when `visited == predicted_reachable` — with the spec's empty-layer criterion kept as an assert/backstop and as the real criterion for `n ≤ 4` (Klein-four exception) and degenerate inputs. This eliminates the expensive "certify an entire empty final layer" problem.
3. **Disk streaming is refuted; recompute-over-reread wins.** Table generation costs ~3 modular add/sub updates per entry (~5 GB/s/core effective); any disk delivers ≤ 0.5 GB/s total. Regenerating the n=50 table (~10 s on 96 cores) beats reading 83 GB from disk (~3 min) every time, and the table doesn't fit Kaggle disk regardless.

## 1. Canonical indexing and combinatorial substrate

- **Canonical partition order (frozen, versioned):** descending-part tuples in reverse-lexicographic order, `(n)` first, `(1^n)` last, for every `m ≤ n` (intermediate levels need indices too). The explicit list and a SHA-256 of its encoding are embedded in every artifact (JSON output, checkpoint, fixture) per spec §19.3.
- **Ranking without hashing:** precompute `Cnt[m][l]` = #partitions of `m` with parts ≤ `l` (an `(n+1)²` u64 table, exact). Rank/unrank of a partition is `O(#parts)` arithmetic. This gives deterministic dense indices at every level `m ≤ n` — no hash maps in any hot path.
- **Partition representation in kernels:** beta-set bitmask (first-column hook lengths as a bitset), `2n ≤ 100` bits → two `u64` words for `n ≤ 50`. All rim-hook enumeration is bit arithmetic on this mask (§2.1).
- **Per-class data:** `z_λ`, `|C_λ| = n!/z_λ` (bigint, then residues), `sgn(λ) = (−1)^{n−ℓ}`, transpose-partition involution `ρ ↔ ρ′` (computed once, stored as an index permutation).
- Distances stored as `u16` (diameters are `O(n)`; assert `< 65535`).

## 2. Murnaghan–Nakayama evaluator

### 2.1 Level operators

For each hook length `l` and level `m`, define the sparse operator

```
M_l : Z^{p(m)} → Z^{p(m+l)},    (M_l v)[ρ] = Σ_{h: l-rim-hook of ρ}  (−1)^{ht(h)−1} · v[ρ \ h]
```

MN says: for a class `ν = (l₁ ≥ l₂ ≥ … ≥ l_k)`, the **full character column** over all `ρ ⊢ n` is

```
X[·, ν] = M_{l₁} ∘ M_{l₂} ∘ … ∘ M_{l_k} (e_∅)
```

(removal order is irrelevant to the value; we fix "apply smallest part first" to enable suffix sharing). The intermediate vector after applying parts `{l_k, …, l_j}` is exactly the character column of class `{l_k,…,l_j}` in `S_{l_j+…+l_k}` — well-defined and reusable.

**Rim-hook enumeration via beta-sets.** `ρ \ h` for an `l`-hook ↔ replace beta number `b ∈ B` by `b−l ∉ B`; addition ↔ `b → b+l`. Height sign: `(−1)^{|B ∩ (b, b+l)|}` — one masked popcount. Each operator row (gather form: for target `ρ`, sum over its removable `l`-hooks) has ≤ `min(ℓ(ρ), ρ₁) ≤ √(m+l)` entries.

**Exact nnz formula (proved, and verified on small cases):** the number of addable `l`-hooks of `μ` equals `l + #removable l-hooks(μ)` (telescoping the beta-set indicator over a window of width `l`). Hence

```
nnz(M_l : m → m+l) = l · ( p(m) + p(m−l) + p(m−2l) + … )        (exact)
```

Example check, `n=5, l=2`: `2·(p(3)+p(1)) = 8`, matching direct enumeration of domino removals over all partitions of 5. This formula lets the planner compute the *exact* generation work for any `n` before running (see §8 and the auto-budget requirement).

**Precompiled CSR operators.** All operators `M_l : (m−l) → m` that any run can touch are materialized once as CSR (gather orientation: rows = level-`m` partitions, entries = `(source_index, sign)` with sign folded as `p−x` arithmetic). Total CSR entries `Σ_{m,l} nnz ≈ 8×10⁶` at `n=45` (~64 MB with u32 indices) — trivial. After this, **the hot path contains no partition manipulation at all**, only indexed adds/subs mod p. The same CSR operators drive the bigint exact-column path.

### 2.2 Suffix-sharing column DP over the trie — adopted, with corrected accounting

Organize all `q` target columns as a trie over *increasing* part sequences: a node = multiset `μ` of parts already applied (sum `m`), extendable iff `n−m = 0` or `n−m ≥ max(μ)`; node payload = the level vector `v ∈ Z_p^{p(m)}`; an edge appends part `l ≥ max(μ)` and applies CSR `M_l`. A DFS with one vector per level visits every node once; a leaf (sum = n) emits one finished column.

**Honest evaluation of the sharing:** total work = `Σ_edges nnz(M_l at that edge)`. The final edges (level `n−l₁ → n`) are one-per-leaf and *cannot* be shared — and they dominate. Using the exact nnz formula:

```
W_top(n) = Σ_{l=1..n}  p_{≤l}(n−l) · l·(p(n−l) + p(n−2l) + …)
```

(`p_{≤l}(n−l)` = #leaves with largest part `l`). Numerically `W_top(n) ≈ (1–1.5)·p(n)²` (e.g. `W_top(5) = 49 = p(5)²` exactly), and lower levels add a comparable amount, truncated by extendability (`#nodes at level m = p_{≤ n−m}(m)`, which collapses near the top). Total:

```
T_gen(n) ≈ (2–4) · p(n)²   modular add/sub updates      [compute exactly via the formula at plan time]
```

So the DP is **near output-optimal** (~2–4 updates per table entry); suffix sharing contributes a modest constant factor (it removes lower-level recomputation), and its real value is that it makes the *exact bigint* column batches cheap (§6.3). Adopt.

**Concrete generation work (updates, per prime; both primes can share one traversal at ~1.3× cost):**

| n | p(n)² | T_gen ≈ 3·p(n)² | 96-core wall (3–6 ns/update, 60–80 % eff.) |
|---|---|---|---|
| 30 | 3.1e7 | 9.4e7 | < 0.1 s |
| 40 | 1.39e9 | 4.2e9 | ~0.3–1 s |
| 45 | 7.9e9 | 2.4e10 | ~1.5–3 s |
| 50 | 4.2e10 | 1.25e11 | ~8–15 s |

**Memory during DFS:** one vector per level on the current path, `Σ_m p(m) ≈ 6·p(n)` residues ≈ 5 MB/task at n=50. Negligible.

**Rayon parallelization (two-level):**
- *Intra-edge:* the dominant final edges are single SpMVs with `p(n)` gather rows — `par_chunks` over target rows (gather form has no write conflicts). This alone parallelizes ≥ 80 % of the work.
- *Inter-subtree:* `rayon::join` over children at the top 2–3 trie levels (each branch clones its ≤ few-hundred-KB path vector). Work-stealing balances the skewed subtree sizes.
- Output is written directly into the stored table's column slot (column index known from the leaf's rank) — generation order need not match index order.

**Rejected alternatives:** (a) *per-row generation* (fix `ρ`, recurse over `ν`) has no analogous shared substructure (the recursion branches on `ρ`-shapes, keyed by (shape, remaining cycle multiset) — a memo table of size ~`Σ p(m)²` entries with hashing in the hot loop and hostile cache behavior); (b) *memoized point evaluation* per `(ρ,ν)` — same defects; (c) column DP also matches the consumption order (the transform needs `X` column-by-target). Column DP is the right choice on all three axes.

### 2.3 The three evaluator modes

| Mode | What | Arithmetic | Cost |
|---|---|---|---|
| (a) Full table, all `(ρ,ν)` | trie DFS, all leaves | mod p (u32), add/sub only — *MN needs no multiplications* | `T_gen(n)` above |
| (b) Single/batched columns on demand | trie restricted to the requested leaf set (shared prefixes kept) | bigint (values ≤ f_max ≤ √(n!) ≈ 2 limbs at n ≤ 50) | per column ≈ top-edge nnz ≈ `T_gen/q ≈ 3q` two-limb ops → ~1–3 ms at n=45 |
| (c) Base-class columns for eigenvalues | mode (b), one column per base class λ_j | bigint, then `ω_ρ = |C|·χ/f` with exact division + divisibility assert | < 1 s total |

Degrees `f_ρ`: hook-length formula, exact bigint (`n!` is only 215 bits at n=50), with a cross-check against mode (b)'s column at `ν = (1^n)` (whose entries are the degrees) — two independent computations of the same data, used as a permanent self-test.

## 3. Modular kernel

### 3.1 Primes

- **Size:** primes in `(2^29, 2^30)`, chosen deterministically (largest below 2^30: 1073741789, 1073741783, …), recorded in all metadata. `p > n` (hence `p ∤ n!`) is trivially satisfied.
- **Why 30-bit, not 31-bit:** with `x, w ≤ p−1 < 2^30`, products are `< 2^60` and a `u64` accumulator holds **16** products before reduction (2^64/2^60). With 31-bit primes only 4 fit — reduction frequency ×4. The 30-bit choice is also the GPU-friendly shape (u32 residues, u64 accumulators, fixed block-16 reduction).
- **How many:** `k = 2` default. Analysis: a positive coefficient vanishing mod one random 30-bit prime has heuristic probability `~2^−29.5`; expected spurious-certification events per run ≈ `q · diam · m · 2^{−29.5·k}`. For n=45, m=100, diam 60: `k=1` → ~0.75 expected events per batch (harmless — auto-repaired, §6.4 — but each costs a certification and a repair pass); `k=2` → ~10⁻⁹. Correctness is independent of `k` (every accepted zero is certified exactly); `k` buys only performance predictability. RAM cost of `k=2` at n=45 is 32 GB — cheap insurance. `k=3` available by config for regime A.
- **Screening shortcut:** `n!` is invertible mod p, so `a_r ≡ 0 ⇔ N_r ≡ 0`; the screen never divides. (`a_r mod p = N_r · (n!)^{-1}` is still computed for the word-count invariant, §11.)

### 3.2 Inner-loop arithmetic — proven bounds

Two provably safe accumulation schemes; pick by microbenchmark, both implemented behind one function:

1. **Lazy-16 u64 (recommended, SIMD-friendly):** accumulate 16 products `< 2^60` into u64 (max `16·(2^30−1)² < 2^64` ✓), then one Barrett reduction: `μ = ⌊2^64/p⌋` precomputed; `t = mulhi64(acc, μ); r = acc − t·p; if r ≥ p { r −= p }` (≤ 2 corrections; standard proof from `μ` error `< 1`). Amortized ~0.3–0.5 cycles/term of reduction overhead. Vectorizes with AVX2 `vpmuludq` (4 lanes of 32×32→64).
2. **u128 accumulation (simplest, provably safe for any row length):** add u64 products into u128 (`add`/`adc`, 2 µops); safe for `R < 2^64` terms trivially; one reduction (`u128 % p` via two-word Barrett) per dot product. Often wins for the memory-bound single-union GEMV.

Montgomery representation is **rejected** for the transform: the inner loop multiplies a *streamed table value* by a *cached weight* once — there are no repeated multiplication chains to amortize a domain conversion. Plain representation everywhere; Barrett for the few standalone mulmods (power updates: `q·m·k` mulmods per radius — negligible).

**Table-generation kernel** needs only mod-p add/sub with sign (`x` or `p−x`) — no multiplications (MN coefficients are ±1). Weighting by `f_ρ` happens once per radius in `W`, not in the table.

### 3.3 Layout and blocking

- **Table:** column-major by target `ν` (each column = `R½` contiguous u32), grouped **even-sign targets first, odd-sign targets second** (enables the `W±` split, §4, and parity filtering by slicing). One array per prime. Written in place during generation.
- **Weights `W`:** `R½ × m` row-major (`ρ`-major) so the inner union loop is unit-stride.
- **GEMM `N = Xᵀ·W`:** parallel over target-column blocks (rayon `par_chunks` of e.g. 64 columns); per column, walk `ρ` in tiles of 16, updating `m` u64 accumulators held in L1 (`m ≤ ~128` per tile; larger `m` is tiled). The table is streamed exactly once per radius (per prime); `W` (≤ 45 MB at n=45, m=100) stays cache/LLC-resident.
- **Bandwidth reality check:** single-union GEMV is memory-bound: n=45 half-table scan = 15.9 GB/prime → ~80–110 ms at 150–200 GB/s. A 100-union batch converts this to compute-bound GEMM (table read amortized 100×) — this is why union batching is the single most important throughput lever. Adopt "batch unions as columns" unchanged.

## 4. Symmetries, active rows, radius-0

### 4.1 Active rows — exactness caveat

`R = ∪_j {ρ : χ^ρ(λ_j) ≠ 0}` computed from the **exact** base columns (mode (c)), never from residues (`f_ρ` or `θ_ρ` can be ≡ 0 mod a 30-bit prime while nonzero — a residue-based test would silently drop a live row). Per-union additional drops where exact `θ_ρ(U) = 0`, valid for `r ≥ 1` only. `R` is automatically closed under transposition (`χ^{ρ′}(λ) = ±χ^ρ(λ)` vanish together). For hook-sparse classes (e.g. `n`-cycles: `R = n` hook rows) this collapses the transform; for the typical bounded classes (`[3]`, `[2,2
`]`) columns are dense and `R ≈ q` — the design must not assume restriction helps; it computes `|R|` and restricts only when it does.

### 4.2 Transpose pairing — exact bookkeeping (mixed-parity safe)

Store only representative rows (say `ρ ≥ ρ′` in canonical order) plus self-transpose rows: `R½ = (q + s)/2`, `s` = #self-conjugate partitions (tiny: partitions into distinct odd parts). Per union maintain **two** power vectors per prime:

```
θ_even[ρ] = Σ_{λ_j even} ω_ρ(λ_j)      θ_odd[ρ] = Σ_{λ_j odd} ω_ρ(λ_j)
P [ρ] ← P [ρ] · (θ_even[ρ] + θ_odd[ρ])   mod p     // θ_ρ^r
P′[ρ] ← P′[ρ] · (θ_even[ρ] − θ_odd[ρ])   mod p     // θ_{ρ′}^r  (spec §11 decomposition)
```

Then build two weight vectors per radius:

```
paired rep ρ:   W₊[ρ] = f_ρ·(P[ρ] + P′[ρ]),   W₋[ρ] = f_ρ·(P[ρ] − P′[ρ])
self-transpose: W₊[ρ] = W₋[ρ] = f_ρ·P[ρ]        // NOT f·(P+P′) — that double-counts
```

and `N_r(ν) = Σ_{ρ ∈ reps ∪ selfs} X[ρ,ν] · W_{sgn(ν)}[ρ]`. Notes: (i) for self-transpose rows `θ_odd = 0` automatically (self-conjugate characters vanish on odd classes) so `P′ = P` — assert this as a runtime invariant; (ii) their contribution to odd targets is zero via `χ = 0`, so the uniform formula is correct; (iii) the target-parity grouping of columns (§3.3) means each GEMM call binds exactly one of `W₊/W₋`. Effect: **table storage, generation output, and GEMM all halve**; power updates double (negligible). Generation cost itself is *not* halved (the DP runs over all intermediate partitions regardless); only `R½` rows are written out.

### 4.3 Parity filtering and radius 0

- Single-parity unions (sign `ε`): at radius `r` only targets with `sgn(ν) = ε^r` are computed (slice one parity block) — another ~2× on the GEMM, and it halves certification volume (§6.2).
- Radius 0: identity type initialized to distance 0 directly; spectral loop starts at `r = 1` (restricted rows are invalid at `r = 0` — spec §7). The full-row radius-0 identity test lives only in the validation suite.

## 5. Exact integer strategy

**Crate choice:** `num-bigint` (MIT/Apache-2.0, pure Rust, Windows-clean) as the default, behind a thin internal alias/trait so a feature flag can swap in `ibig` (MIT/Apache, ~2–10× faster mid-size, but low maintenance activity — acceptable as opt-in only). `malachite` is **rejected as default**: LGPL-3.0 is awkward for statically linked PyO3 wheels distributed to Kaggle (relink obligation), even though its performance is best-in-class. `rug`/GMP rejected per the Windows constraint. This is safe because *all bigint work is off the hot path*: profiling target is < 10 % of wall time, and the sizes below show schoolbook multiplication suffices.

**Where bigints appear, with sizes (`L` = 64-bit limbs):**

| Quantity | Bound | Size at n=45–50 |
|---|---|---|
| `n!` | — | 205–215 bits (4 L) |
| `f_ρ` (hook formula), `χ^ρ(ν)` | ≤ √(n!) | ≤ 110 bits (2 L) |
| `ω_ρ(λ) = |C|χ/f` (exact ÷, assert) | ≤ `|C_λ|` | ≤ 215 bits (4 L); for bounded classes `[k…]`: ~`k·log₂n` ≈ 15–25 bits (1 L) |
| `θ_ρ(U)`, `θ_even/odd` | ≤ `|U|` | as above |
| `θ_ρ^r` (incremental per radius) | `r·log₂|U|` | bounded classes: ~25·60 = 1.5 kbit (25 L); worst case (huge classes): 215·75 ≈ 16 kbit (250 L) |
| `W_r = f·θ^r`, cert dot result, `a_r ≤ |U|^r` | same order | same |

**Exact powers are maintained incrementally** (`power *= θ` per radius, per active union, over active rows), matching the modular power update. Cost: `q` mults of (≤4 L)×(growing) per union-radius — bounded classes: ~5–10 s *total* for a 100-union, 60-radius run at n=50; worst-case classes: ~25–50 s total. Memory: `m·q·avg_limbs·8 B` — 5 GB (bounded) to ~40 GB (worst) at n=50/m=100; if tight, the alternative is recomputing `θ^{r−2}` by binary exponentiation per certification radius (documented trade, config switch). Exact `θ` values also produce the mod-p `θ` by reduction — one source of truth.

**CRT is rejected as the primary certification mechanism:** certifying a zero needs `Π p_i > |U|^r` → `r·log₂|U|/30` primes (e.g. 19 primes for `[3]` at r=40, 200+ for large classes), each requiring restricted column generation mod a fresh prime. The direct bigint dot (§6.3) is simpler and faster at these sizes. CRT remains an optional path only for small-`r` full-count reconstruction (out of scope for the support-only output).

## 6. Certification: the exact-fallback subsystem

### 6.1 The monotonicity lemma that makes exactness affordable

**Lemma.** For any nonempty `U` closed under inversion (every class is), `a_{r+2}(ν) ≥ a_r(ν)`: append `u·u^{−1}` for a fixed `u ∈ U` — an injection of length-`r` words into length-`r+2` words with the same product.

**Consequence.** Within each parity chain (`r` even / odd), positivity is monotone. Therefore `a_{d−2}(ν) = 0` certifies `a_{r′}(ν) = 0` for *all* `r′ ≤ d−2` in that chain, and `a_{d−1}(ν) = 0` covers the other chain. So a claimed distance `d(ν) = d` is fully certified by **at most two exact zero-evaluations** — and:

- single-parity unions: the `d−1` chain is zero *by parity* (theory, invariant 9.6) → **one** evaluation;
- `d ≤ 2`: `a_0` (identity indicator) and `a_1` (= input weights, invariant 9.2) are exact by theory → **zero** evaluations;
- unreachable types (odd types under even unions): zero by parity/normal-subgroup theory → **zero** evaluations.

Total mandatory exact work per (union): ≈ 1–2 evaluations per reachable type. This is the true "fallback frequency" — driven by structure, not by prime collisions. (Prime collisions add `~q·diam·2^{−29.5k}` extra evaluations — nil for `k = 2`.)

### 6.2 Layer gate and termination

- Distance assignments at radius `r` (first nonzero residue in any prime — a *proof* of positivity) are **provisional** until their `d−1/d−2` certifications complete. Certifications run asynchronously on a work queue; they never block the radius loop.
- **Termination:** predicted reachable set (§0, item 2) — `A_n`-types for even-only unions, all types for mixed/odd (n ≥ 5; `n ≤ 4` and identity/empty inputs go to the brute-force path). Run ends when `visited == predicted` **and** the certification queue is drained with no pending repairs. The spec's "empty `new_r`" test runs as an assert; if it fires before the predicted set is covered, that is a hard error (or the n≤4 path).
- Safety bound: abort with diagnostics if `r` exceeds a configured multiple of the expected diameter (default `4n`).

### 6.3 Cost of one certification, and batching

`a_r(ν) = (Σ_{ρ∈R_active} f_ρ·χ^ρ(ν)·θ_ρ^r)/n!` (assert exact division, assert ≥ 0):

- **Exact column `χ^{·}(ν)`** — depends only on `ν`: computed by the bigint trie path (mode (b), §2.3), **shared across all unions and both radii** needing it. Cached per-`ν` (LRU; full exact-table caching is impossible: 127 GB at n=45). All columns needed over a whole run ≈ one exact table generation ≈ `3q²` two-limb ops → 10–30 s at n=45, ~2–4 min at n=50, spread across the run, rayon-parallel.
- **Dot with `W_{r}` (exact, cached per union-radius):** `|R_active|` mults of (2 L)×(`θ^r` limbs). Bounded classes at n=45: ~89 k × 15 ns ≈ **1.3 ms/certification**; n=50: ~3 ms. Large classes (250 L): ~10× more.

**Totals** (m = 100 unions, dots are per-(type, union), columns shared):

| n | class size regime | cert wall (96 cores) |
|---|---|---|
| 40 | bounded | ~25–40 s |
| 45 | bounded | ~2–3 min |
| 50 | bounded | ~11–15 min |
| 50 | huge classes | ~1.5–2 h → **must checkpoint mid-certification; flagged as the n=50 risk item** |

Batch discipline: queue entries keyed `(ν, r, union)`; scheduler groups by `ν` (compute column once, evaluate all pending unions), and per radius the layer's column set runs as one restricted-trie batch (prefix sharing again).

### 6.4 Repair path (false modular zero)

If a certification returns `a_{d−2}(ν) > 0` (probability ~2^{−59} per event at k=2, but the code path is mandatory for exactness): set `d(ν) ← d−2`, enqueue certification at `d−4` (and `d−3` if mixed parity), loop until a certified zero or `d ≤ 2`. Distances of other types are unaffected (per-type independence). Emit a metadata counter (`repairs: 0` expected) into the output JSON.

## 7. n ≥ 50 strategy: `H_α` streaming vs stored vs sweep vs disk

**Decision inputs, computed exactly and cheaply up front:** the exact eigenvalue list `θ_ρ(U)` (from base columns) → `|E_U|` = #distinct values by sort-dedupe; RAM/disk budgets; `m`.

1. **Eigenvalue grouping `H_α` — adopted as the primary n ≥ 50 path for the primary use case.** For bounded-support classes the eigenvalues live in a polynomial range: transpositions `ω_ρ = Σ contents(ρ)` ∈ `[−C(n,2), C(n,2)]` → `|E| ≤ 2451` at n=50 vs `R½ ≈ 102 k` rows (**40×+ compression, guaranteed by range**); `[3]`: `|E| ≤ 2|C|+1 ≈ 8×10⁴` (mild); `[2,2]` and unions: range `Σ|C_j|` may exceed `R` — **measure, don't assume**. Aggregation `H_α(ν) = Σ_{θ_ρ=α} f_ρ·χ^ρ(ν) mod p` is built in **one** generation sweep (table never stored): per emitted column, `R½` bucket-adds per union → aggregation for a wave of `m_w` unions ≈ one dense radius of GEMM. Afterwards every radius costs `|E|·q` MACs (n=50, `|E|`=2451: **~7 ms/radius**) plus `|E|` mulmods for `α^r`. Memory: `|E|·q·4 B·k` per union → waves sized to RAM (e.g. 20 unions/wave = 40 GB at `|E|`=2451). Exact division by `n!` only at the end of each radius sum (the `H_α/n!` are *not* individually integral — spec §15); certification unchanged (bigint columns, independent of storage). Transpose pairing composes: keep `H` over representative rows with the `W±`-style split (bucket by the *pair* `(θ_ρ, θ_{ρ′})`).
2. **Fallback when `|E_U| ≳ R½/4`: fused all-radii sweep.** All powers `θ^r mod p, r = 1..r_max` are known up front, so one generation sweep can dot every emitted column against all `r_max` weight vectors at once: cost `R½·q·r_max·m` MACs — n=50, single union, `r_max=75`, 2 primes: ~22 s total; m=100: ~37 min; result memory `q·r_max·4 B` = 49 MB per (prime, union). Overshoot in `r_max` is linear and cheap; a shortfall triggers a second sweep. This is within ~2–4× of stored-table mode in total MACs (identical FLOPs; stored mode wins only via per-radius adaptivity — early union exit, parity slicing).
3. **Stored table remains feasible at exactly n=50** with pairing + k=2: 167 GB in a 300+ GB VM. Offered as regime B when RAM verifies at startup; automatic fallback to (1)/(2) otherwise.
4. **Disk streaming: refuted.** 83 GB/prime exceeds Kaggle's ~70 GB disk; even if it fit, ≤ 0.5 GB/s disk vs ~10 s in-RAM regeneration makes rereading strictly worse. *Recompute-don't-reread* is adopted globally (checkpoints also regenerate tables, §10).

**Recommendation:** planner rule — `if table fits RAM → stored; elif |E_U|·q·k·wave ≤ RAM and |E_U| < R½/4 → H_α; else → fused sweep`. All three share the generation sweep, classification, and certification code.

## 9. GPU-readiness: the transform backend trait

Everything the math layer needs is one exact modular GEMM against a resident/streamed table. Documentation-level contract:

```rust
/// Exact modular transform backend for one (n, prime) context.
/// All residues u32 in [0, p), PLAIN representation (not Montgomery).
pub trait ModTransformBackend: Send + Sync {
    /// Ingest table columns (canonical target order, representative rows only,
    /// even-sign targets then odd-sign). Backend may store device-side or mirror host.
    /// Called once per (n, prime, run); columns arrive in panels ≥ 64 MB.
    fn load_table(&mut self, meta: TableMeta, cols: &mut dyn ColumnPanelStream) -> Result<TableId>;

    /// out[t][j] = Σ_ρ table[ρ, targets.start+t] · w[ρ·m + j]  (mod p), exact.
    /// w: R½ × m row-major (ρ-major); out: |targets| × m; targets is a
    /// parity-aligned slice. Tiling is backend's choice; accumulation MUST be
    /// provably overflow-free (reference contract: p < 2^30, u64 accum,
    /// reduction period ≤ 16 — the documented proof carries to CUDA u64 math
    /// and to wgpu via 2×u32 emulation or f64 with block ≤ 2^53/p²).
    fn gemm(&self, t: TableId, w: &[u32], m: usize, targets: Range<u32>, out: &mut [u32]) -> Result<()>;

    /// Optional fused multi-radius form for streamed/no-store regimes:
    /// w_multi packs r_range.len()·m weight vectors; default = loop over gemm.
    fn gemm_multi(&self, ...) -> Result<()> { ... }
}
```

Contract points: (i) **bitwise-exact residues** — CPU rayon reference and any GPU backend must agree exactly, giving a free cross-backend test; (ii) transfer granularity: table upload once per (n, prime) (n=40: 2.8 GB — fits any relevant GPU; larger n: streamed panels with per-panel `gemm` calls driven by the host, reused across all m unions before eviction); weights up per radius (`R½·m·4 B`), results down per radius (`q·m·4 B`); (iii) classification, distances, certification, and all bigint work stay host-side; (iv) the backend never sees partitions — only dense integer blocks and the parity split boundary.

## 10. Checkpointing (numerics-relevant state)

Serialized per (n, union-batch): version + config hash + partition-list hash + prime list; per union: radius `r`, `visited` bitset, `distance: [u16]`, modular power vectors `P, P′` per prime, exact `θ` values, exact power vectors (or a flag to rebuild by binary exponentiation), certification queue (pending `(ν, r, union)` triples + completed count), repair log. **Tables, CSR operators, and `H_α` blocks are not checkpointed** — regenerated on resume (§7, seconds). For fused-sweep mode: completed column ranges + per-type first-hit radius arrays. Checkpoint cadence: after every radius and every N certifications; sizes are tens of MB (n=45, m=100: powers 36 MB + distances 18 MB + bitsets ~2 MB).

## 11. Numerics test hooks (beyond spec §22)

Per-radius, per-prime runtime guards (all O(q) or O(1)): word-count `Σ_ν |C_ν|·a_r(ν) ≡ |U|^r (mod p)`; `r = 1` indicator exactness; self-transpose `P′ = P` assert; `Σ_ρ f_ρ² = n!` and `Σ_ρ f_ρ·χ^ρ(ν) = 0 (ν ≠ 1^n)` on the generated table (one GEMV); random column-orthogonality spot checks `Σ_ρ χ^ρ(μ)χ^ρ(ν) ≡ z_ν δ_{μν}`; degrees via hook formula vs identity-column DP; unpaired-vs-paired transform equality on sampled columns; brute-force BFS equality for `n ≤ 10`; SymPy fixtures for `χ` at `n ≤ 14`; engineered prime-divisible coefficients (exhaustive small-n search with the exact backend) asserting the certification/repair path fires.

## 12. Risks / open questions

1. **Certification volume at n=50 with large classes** (§6.3) — the one place the 2 h budget genuinely binds; mitigations: checkpointing mid-queue, `ibig` feature, parity halving, per-type single-cert for single-parity unions.
2. `|E_U|` for multi-class unions is unproven — measured per run; both fallbacks exist.
3. Throughput constants (scatter rate, MAC rate on Kaggle vCPUs) are estimates; the planner recalibrates from a built-in microbenchmark.
4. `Σ_m p(m)`-level CSR operator memory grows to ~0.5 GB by n=50 — verify, else build per-level on the fly.
